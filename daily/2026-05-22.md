## What I learned

### 1. Chain-aware context

A normal AI assistant can answer from documents, chat history, and tool outputs. A Web3-aware agent also needs chain-specific facts:

- chain id / network
- contract address
- ABI and method
- block number / timestamp
- balance, allowance, and transaction history
- explorer link or other verifiable citation

The important rule is: the model should not guess onchain facts from memory. It should read them through RPC, explorers, indexers, or documented APIs, and it should keep sources attached.

### 2. Web3 tool use

Web3 tool use is more dangerous than ordinary search or file tools because some tools can change asset balances, approvals, or contract state.

The most useful distinction is read vs write:

- Read tools: query balances, block number, contract view functions, logs, transaction receipts.
- Write tools: generate/sign/send transactions, approve tokens, call contract state-changing methods, deploy contracts.

Read tools can be broader, but write tools need structured parameters, simulation, risk checks, logging, and human confirmation.

### 3. Agent workflow

An AI × Web3 workflow should not be “let the model freely decide every next step.” A safer workflow should look more like a state machine:

1. Load goal and constraints.
2. Read chain-aware context.
3. Draft a plan.
4. Run simulation or dry-run checks where possible.
5. Show risks and expected effects.
6. Wait for user confirmation.
7. Execute only within the approved scope.
8. Verify result with tx hash / explorer link / logs.
9. Record a trace for review.

This makes the agent easier to debug and safer to operate.

### 4. Agent wallet and account abstraction

The biggest shift today: an Agent Wallet is not “give AI a private key.” It is a permission system.

A safer design is to give the agent a limited, revocable, auditable action space:

- limited chain id
- allowed contract addresses
- allowed methods
- spending or value limit
- expiry time
- max number of operations
- pause / revoke path
- logs and explorer evidence

Account abstraction and smart accounts make this design more realistic because the account itself can express rules such as session keys, daily limits, social recovery, multi-signature approval, gas sponsorship, or method-level permissions.

## My Week 1 synthesis

My Week 1 understanding is:

AI brings planning, summarization, tool orchestration, and coding assistance.
Web3 brings wallets, signatures, contracts, transactions, public state, and irreversible execution.

The overlap is powerful but risky. If an agent can touch Web3 tools, the design must separate:

- explanation vs execution
- read tools vs write tools
- public context vs secrets
- draft transaction vs signed transaction
- model suggestion vs user authorization
- AI summary vs verifiable onchain evidence

The core safety principle I want to keep using is:

AI may prepare, explain, simulate, and verify. Humans or explicitly constrained wallet policies must authorize risky execution.

## Key output

Task note:

- `tasks/week2-agent-wallet-permission-strategy.md`

核心结论：

1. Agent payment 不只是“自动付款”，而是“在预算、权限、确认和日志边界内付款”。
2. 低风险动作可以自动化，例如搜索、报价比较、读取余额、保存 receipt。
3. 高风险动作必须人工确认，例如首次付款给新服务、超额付款、合约 approval、policy 修改、未知合约调用。
4. Policy / guard 层是自然语言 agent 和不可逆链上执行之间的安全边界。
5. ERC-4337、Safe、guard / policy 的价值不是让 agent 更“自由”，而是让 agent 的行动更可限制、可撤销、可审计。

## Automation vs confirmation summary

| Action | Decision |
| --- | --- |
| Search public sources | Automate |
| Compare paid services | Automate |
| Set budget | Human confirmation |
| First payment to new service | Human confirmation |
| Small allowlisted payment under limit | Automate |
| Unlimited approval | Deny by default |
| Contract / module / policy change | Human confirmation |
| Save receipt and log | Automate |
| WCB submission | Human confirmation |

## Reflection

昨天我选择的 Week 2 主线是 Payment / Commerce / Settlement。今天的学习让我意识到：Agent commerce 的难点不是“能不能付钱”，而是“怎么让付款不变成失控授权”。

一个可用的 agent commerce flow 至少需要三层：

1. Commerce layer: quote, order, delivery, acceptance, refund / dispute.
2. Wallet layer: budget, payment, receipt, revocation.
3. Policy layer: allowlist, spending limit, simulation, human confirmation, logs.

如果没有 wallet / policy 层，agent 自主支付更像是危险的自动化；如果有清晰边界，它才可能成为可验证、可审计的 Web3 workflow。

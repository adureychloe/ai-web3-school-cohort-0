## Key understanding

今天最重要的理解是：

> Agent commerce 的最小闭环不是“agent 自动付款”，而是“intent → quote → policy → payment → delivery → acceptance → proof”。

几个具体结论：

1. x402 适合做付费 API / agent endpoint 的小额支付入口，但它只解决付款协议，不解决全部信任和验收问题。
2. EIP-3009 可以让付款像“签一次受限授权”，但仍然必须被金额、deadline、nonce、token、收款方和 policy 限制。
3. ERC-8004 可以帮助 agent 发现服务、检查身份和 reputation，但不能替代用户判断。
4. Wallet policy / guard 是安全边界：它决定哪些付款可以自动执行，哪些必须人工确认。
5. Proof log 很关键，因为 agent commerce 需要向用户解释：为什么付款、付给谁、拿到了什么、是否符合预算。

## Flow summary

```text
User intent
  → Agent plan
  → Service discovery
  → Quote / HTTP 402 payment requirement
  → Wallet policy check
  → Human review if needed
  → x402 / EIP-3009 payment
  → Service delivery
  → Acceptance
  → Receipt / proof log
  → Refund or dispute if failed
```

## Automation vs confirmation

| Step | Default decision |
| --- | --- |
| Public research | Automate |
| Service discovery | Automate, but verify identity / reputation |
| Quote comparison | Automate |
| Budget setup | Human confirmation |
| New service first payment | Human confirmation |
| Allowlisted small fixed-price payment | Can automate under policy |
| Approval / delegation / module install | Human confirmation |
| Subjective quality acceptance | Human review |
| Receipt and proof logging | Automate |
| WCB evidence submission | Human confirmation |

## Reflection

这次学习让我把“AI × Web3 支付”从一个很抽象的方向拆成了具体流程。之前我容易把重点放在“agent 能不能自己付款”；现在更清楚的是：真正有价值的问题是如何让付款可限制、可解释、可恢复、可审计。

对 Hackathon 或后续项目来说，一个可行的 MVP 不是一开始就接真实资金，而是先做一个模拟版：

- 模拟 402 payment required。
- 让 agent 生成付款意图。
- policy engine 判断是否允许。
- 超过阈值时要求用户确认。
- 生成 receipt / proof log。

这样可以先验证产品逻辑和安全边界，再考虑接真实 x402、wallet 或 CAW。

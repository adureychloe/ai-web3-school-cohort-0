## What I learned

### 1. EOA is simple but coarse-grained

An EOA is controlled by a private key. It can initiate transactions directly, but the permission model is basically all-or-nothing: whoever can sign can act.

This makes EOA easy for personal learning and testnet experiments, but dangerous if:

- the seed phrase is exposed
- the user signs a malicious message
- an automation tool receives too much authority
- the user cannot clearly understand what the transaction does

For AI × Web3, my rule is: an agent can explain or prepare an EOA transaction, but it should never receive the EOA private key or seed phrase.

### 2. Smart account turns account control into programmable rules

Account abstraction / smart accounts make the account itself more programmable.

Useful rule types include:

- allowed contracts
- allowed methods
- spending limits
- expiry time
- session keys
- recovery rules
- gas sponsorship
- revoke / pause paths

This matters for AI agents because the goal should not be “give the AI my wallet.” The better design is: define a narrow, revocable, auditable permission space where the agent can help.

### 3. Multisig distributes authority

A multisig wallet requires multiple approvals, such as 2-of-3 or 3-of-5.

It is slower than an EOA, but it reduces single-key risk and is useful for:

- team funds
- DAO treasuries
- protocol admin actions
- high-value transactions
- decisions where one person should not have unilateral power

For AI-assisted workflows, the AI can draft a multisig proposal or summarize risks, but the signers still review and approve.

## Comparison summary

- EOA: best for simple personal actions; biggest risk is one-key control.
- Smart account: best for limited automation and agent workflows; biggest risk is contract or permission misconfiguration.
- Multisig: best for group control and high-value actions; biggest risk is signer coordination or compromised threshold.

# Hackathon 项目方向讨论 — 2026-06-01

> 当前项目 "Agent Commerce Sandbox"（模拟 x402 + policy + receipt）可能缺乏黑客松新意。
> 需要讨论后决定是否调整方向。

---

## 当前项目的问题

- 太像课程作业：就是把 Week 2 学了的东西做成 CLI demo
- 没有核心洞察：Cobo 官方 Recipe 已经提供了类似功能
- "模拟"本身就是弱点：不接真实资金是安全的选择，但也意味着 demo 缺乏"真东西"
- 在 100+ 参赛作品中，很难脱颖而出

---

## 候选方向 1：Agent 当卖家（中改动量）

**核心翻转**：当前 agent 是消费者（花钱买服务），改为 agent 是生产者（提供服务、收 x402 付款、自动交付）。

### 概念
```
用户 → 发请求 → Agent 返回 402 → 用户付款 →
Agent 验证链上支付 → 自动交付结果 → 输出链上 receipt
```

### 特点
- Cobo 的 x402 Recipe 本来就是面向这个场景的
- 可以用测试网接收真实 x402 付款（0 gas？实际还是需要 gas）
- 做成一个可部署的 AI endpoint 模板
- 比"模拟花钱"更有冲击力

### 改动量
- 中等：改造 engine 流程，加一个小 HTTP server（FastAPI）
- 测试网支付层直接可用（反过来收钱而已）
- 需要加一个验证逻辑：收到 tx 后才交付

---

## 候选方向 2：多 Agent 交易市场（大改动量）

**核心概念**：多个 agent 互相买卖服务。

### 概念
```
Agent A（数据分析服务）  ←→  Agent B（需要数据，从 A 购买）
                                 ↕
                            On-chain receipt + reputation
                                 ↕
                         Policy 依据 reputation 调整信任级别
```

### 特点
- 更接近 "Agentic Economy" 的 vision
- 有故事性和演示效果
- Reputation 系统本身就是有价值的产出

### 改动量
- 大：需要多 agent 协调架构、reputation 系统、排序/发现机制
- 不适合每天 1 小时的节奏

---

## 候选方向 3：Anti-Prompt-Injection Payment Guard（小改动量）

**核心概念**：Prompt injection 导致 agent 被诱导多付钱/付错人是真实存在但没人解决的痛点。

### 概念

Agent 收到服务返回后，policy 引擎检测异常：
```
服务返回 {"price": "0.25", "to": "0xabc..."}
         ↓
Policy 对比：预期价格 vs 实际返回 → 金额突变？
         ↓
        对比：预期收款方 vs 实际 → 收款方不一致？
         ↓
        对比：token / network 是否匹配预期？
         ↓
异常 → 自动拦截 + 生成"攻击报告"
正常 → 放行
```

### 输出
- 攻击报告（JSON + markdown）
- 证据可在链上存证
- 3 个演示场景：正常、金额篡改、收款方替换

### 改动量
- 小：在现有 `policy_checker.py` 基础上扩展
- 新增 `anomaly_detector.py` 模块
- 测试网支付层直接可用（证明能防住真实攻击场景）
- 独立于 payment 模式（sim/real 都支持）

## 最终决策（2026-06-03）

**合并方向① + 方向③ → "Secure Agent Commerce"**

决策理由：
- 方向① 填补 Cobo 生态中的"供应端"空白（Cobo 已有消费端 Recipe，无卖家端 Recipe）
- 方向③ 提供安全层，让 Agent 不仅能赚钱还能安全地赚钱
- 两者合并在一个 Demo 故事线下效果更强：正常赚钱 → 被攻击拦截 → 生成报告上链

实现计划详见 `tasks/merge-merge-plan.md`
